# Harness Engineering: 에이전트 퍼스트 세계에서 Codex 활용하기

> **원문**: [Harness engineering: leveraging Codex in an agent-first world | OpenAI](https://openai.com/index/harness-engineering/) (2026년 2월 11일)  
> **참고 분석**: [InfoQ](https://www.infoq.com/news/2026/02/openai-harness-engineering-codex/), [NxCode Complete Guide](https://www.nxcode.io/resources/news/harness-engineering-complete-guide-ai-agent-codex-2026), [Martin Fowler's Blog - Birgitta Böckeler](https://martinfowler.com/articles/exploring-gen-ai/harness-engineering.html)

## 개요

OpenAI가 **하네스 엔지니어링(Harness Engineering)** 이라는 내부 방법론을 공개했다. 5개월 동안의 내부 실험에서 엔지니어 팀이 AI 에이전트(Codex)를 활용해 **약 100만 줄의 프로덕션 코드를 단 한 줄의 수동 코드 없이** 빌드하고 배포했다.

핵심 통찰: **소프트웨어 팀의 주된 역할이 코드 작성에서 환경 설계, 의도 명세, 피드백 루프 구축으로 전환되고 있다.**

---

## 하네스 엔지니어링이란?

### 정의

> "하네스(Harness)"는 AI 에이전트를 둘러싼 인프라, 제약, 피드백 메커니즘 전체를 의미한다. 에이전트가 신뢰할 수 있게 동작하도록 설계된 환경 그 자체이다.

용어의 어원은 마승구(馬乘具, 말을 다루는 장비)에서 온다:
- **말 (Horse)**: AI 모델 — 강력하지만 방향성이 없음
- **하네스 (Harness)**: 제약, 가드레일, 피드백 루프로 구성된 인프라
- **기수 (Rider)**: 직접 작업하지 않고 방향을 제시하는 인간 엔지니어

### 4가지 핵심 기능

잘 설계된 하네스는 다음 네 가지를 수행한다:

1. **제한 (Constrains)**: 에이전트가 실행할 수 있는 것을 제한 (아키텍처 경계, 의존성 규칙)
2. **정보 제공 (Informs)**: 기대 동작을 에이전트에게 알림 (컨텍스트 엔지니어링, 문서)
3. **검증 (Verifies)**: 출력의 정확성 확인 (테스트, 린팅, CI 검증)
4. **수정 (Corrects)**: 에이전트 실패에 대응 (피드백 루프, 자가 수복 메커니즘)

## OpenAI의 실험 결과

| 항목 | 결과 |
|------|------|
| 기간 | 5개월 |
| 코드 규모 | 100만 줄 이상 |
| 수동 작성 코드 | **0줄** |
| 개발 속도 | 인간 대비 약 1/10 시간 |
| 상태 | 내부 사용자 및 외부 알파 테스터 대상 배포 완료 |
| 운영 | 배포, 장애, 수정 모두 에이전트가 처리 |

엔지니어들은 직접 코드를 작성하는 대신 **풀 리퀘스트와 CI 워크플로를 통해 AI 에이전트를 가이드**했다. 에이전트는 구현, 문서화, CI 설정, 옵저버빌리티, 툴링을 자율적으로 처리했다.

## 3대 핵심 축

### 1축: 컨텍스트 엔지니어링 (Context Engineering)

에이전트가 필요한 시점에 필요한 정보를 갖도록 보장한다.

**정적 컨텍스트:**
- 리포지토리 기반 문서 (아키텍처 명세, API 계약, 스타일 가이드)
- 프로젝트별 인스트럭션 파일 (`AGENTS.md`, `CLAUDE.md`)
- 린터로 검증되는 크로스링크 설계 문서

**동적 컨텍스트:**
- 옵저버빌리티 데이터 (로그, 메트릭, 트레이스)
- 시작 시 디렉토리 구조 매핑
- CI/CD 파이프라인 상태 및 테스트 결과

> **핵심 원칙**: 에이전트의 관점에서 접근할 수 없는 지식은 존재하지 않는 것과 같다. Google Docs, Slack, 또는 사람들의 머릿속에만 있는 문서는 시스템에 보이지 않는다. **"리포지토리가 단일 진실 공급원이어야 한다."**

---

### 2축: 아키텍처 제약 (Architectural Constraints)

전통적인 프롬프팅과 근본적으로 다른 접근. "좋은 코드를 써라"고 지시하는 대신 **하네스가 품질 기준을 기계적으로 강제**한다.

**의존성 레이어링 예시:**

```
Types → Config → Repo → Service → Runtime → UI
```

각 레이어는 왼쪽 레이어에서만 임포트 가능 — 구조적 테스트와 CI 검증으로 강제.

**강제 메커니즘:**
- **결정론적 린터**: 위반을 자동으로 플래그하는 커스텀 규칙
- **LLM 기반 감사자**: 다른 에이전트의 코드를 검토하는 에이전트
- **구조적 테스트**: 아키텍처 규칙의 프로그래밍적 검증
- **프리커밋 훅**: 코드 커밋 전 자동 검사

> **역설적 이점**: 솔루션 공간을 제한하면 오히려 생산성이 향상된다. 제약 없는 에이전트는 막다른 길을 탐색하느라 토큰을 낭비하지만, 경계가 있는 에이전트는 올바른 해답에 더 빠르게 수렴한다.

---

### 3축: 엔트로피 관리 (Entropy Management)

시간이 지남에 따라 AI가 생성한 코드베이스에는 엔트로피가 축적된다 — 문서가 현실과 괴리되고, 네이밍 패턴이 갈라지고, 구식 코드가 쌓인다.

**엔트로피 관리 에이전트:**
- **문서 일관성**: 문서가 현재 코드와 일치하는지 검증
- **제약 위반 스캐너**: 이전 검사를 우회하는 코드 식별
- **패턴 강제**: 확립된 패턴에서 벗어난 것 수정
- **의존성 감사자**: 불필요하거나 순환 의존성 추적 및 해결

이 에이전트들은 일정에 따라(매일, 매주, 이벤트 트리거) 실행되며 코드베이스의 건강을 유지한다.

## Feedforward & Feedback 제어 모델

*Martin Fowler 블로그 (Birgitta Böckeler, 2026.04.02) 기반*

하네스는 두 가지 보완적 메커니즘으로 동작한다:

| 구분 | 가이드 (Feedforward) | 센서 (Feedback) |
|------|---------------------|----------------|
| 시점 | 에이전트 동작 **전** | 에이전트 동작 **후** |
| 목적 | 원치 않는 동작 미리 방지 | 결과 관찰 후 자가 수정 |
| 특징 | 첫 시도에서 품질 출력 확률 증가 | LLM 소비에 최적화 시 가장 강력 |

> "피드백 전용 시스템은 반복적인 실수를 만들고, 피드포워드 전용 시스템은 검증 없이 규칙을 인코딩한다."

### 결정론적 vs. 추론적 제어

**결정론적 제어 (Computational):**
- 빠르고, CPU 기반, 신뢰성 높음
- 예: 테스트, 린터, 타입 체커, 구조적 분석
- 실행 시간: 밀리초~초

**추론적 제어 (Inferential):**
- 의미론적 분석, AI 코드 리뷰, LLM-as-judge
- GPU/NPU 기반, 더 느리고 비용 높음
- 신뢰성: 비결정론적이지만 의미론적으로 풍부

### 품질 유지를 위한 타이밍

| 단계 | 도구 예시 |
|------|----------|
| 통합 전 (빠른 검사) | LSP 통합, 아키텍처 문서, AGENTS.md |
| 초기 자가수정 루프 | 코드 리뷰 에이전트, ESLint, Semgrep, 커버리지 측정 |
| 통합 후 (비용 높은 검사) | 아키텍처 리뷰 스킬, 뮤테이션 테스트, 인간 수동 리뷰 |
| 지속적 모니터링 | 데드코드 감지, 의존성 취약성 스캔, 런타임 SLO 저하 감지 |

## 산업계 구현 사례

### OpenAI: 제로 수동 코드

| 역할 | 전통적 방식 | 하네스 엔지니어링 |
|------|------------|-----------------|
| 코드 작성 | 주요 기능 | **절대 하지 않음** |
| 아키텍처 설계 | 부분적 책임 | 주요 기능 |
| 문서화 | 부차적 고려 | **핵심 인프라** |
| 코드 리뷰 | 표준 코드 리뷰 | 출력 + 하네스 효과성 리뷰 |
| 디버깅 | 코드 읽기 | 에이전트 행동 패턴 분석 |
| 테스트 | 테스트 스위트 작성 | 에이전트를 위한 테스트 전략 설계 |

### Stripe: 미니언(Minions) 대규모 운영

매주 1,000개 이상의 풀 리퀘스트를 병합하는 워크플로:

1. 개발자가 Slack에 태스크 게시
2. Minion이 코드 생성
3. Minion이 CI/CD 통과
4. Minion이 풀 리퀘스트 생성
5. 인간이 리뷰 후 병합

2~4단계에서 개발자 개입 **없음** — 테스트 실행, CI 검증, 스타일 준수, 문서 업데이트를 하네스가 처리.

### LangChain: 미들웨어 퍼스트 아키텍처

하네스를 조합 가능한 미들웨어 레이어로 구성:

```
Agent Request
  → LocalContextMiddleware (코드베이스 매핑)
  → LoopDetectionMiddleware (반복 방지)
  → ReasoningSandwichMiddleware (연산 최적화)
  → PreCompletionChecklistMiddleware (검증 강제)
  → Agent Response
```

각 레이어는 핵심 에이전트 로직을 수정하지 않고 기능을 추가. 테스트 가능하고 진화 가능한 시스템.

> **LangChain 결과**: 하네스만 변경(모델 미변경)으로 Terminal Bench 2.0에서 52.8% → 66.5% (Top 30 → Top 5) 성능 향상.

## 실전 적용 프레임워크

### Level 1: 기본 하네스 (개인 개발자)

**Claude Code, Cursor, Codex 사용자 설정:**
- `CLAUDE.md` 또는 `.cursorrules`에 프로젝트 컨벤션 문서화
- 린팅/포매팅을 위한 프리커밋 훅
- 에이전트 자가검증 가능한 테스트 스위트
- 일관된 네이밍의 명확한 디렉토리 구조

**투자**: 1~2시간 | **효과**: 일반적인 에이전트 실수 방지

### Level 2: 팀 하네스 (3~10명 팀)

**Level 1 추가 사항:**
- 팀 전체 컨벤션 인코딩한 `AGENTS.md`
- CI로 강제되는 아키텍처 제약
- 반복 태스크를 위한 공유 프롬프트 템플릿
- 린터 검증되는 문서-as-코드
- AI 생성 풀 리퀘스트용 코드 리뷰 체크리스트

**투자**: 1~2일 | **효과**: 팀 전체에서 일관된 에이전트 동작

### Level 3: 프로덕션 하네스 (대규모 조직)

**Level 2 추가 사항:**
- 커스텀 미들웨어 (루프 감지, 추론 최적화)
- 에이전트가 로그와 메트릭을 읽을 수 있는 옵저버빌리티 통합
- 주기적 실행 엔트로피 관리 에이전트
- 하네스 버전 관리 및 실험
- 에이전트 성능 모니터링 대시보드
- 막힌 에이전트를 위한 에스컬레이션 정책

**투자**: 1~2주 | **효과**: 에이전트가 자율적 기여자로 기능

## 흔한 실수와 함정

### 1. 제어 흐름 과도 설계
모델은 빠르게 발전한다. 어제의 복잡한 파이프라인이 오늘 모델에는 불필요할 수 있다. 모델이 충분히 발전하면 제거할 수 있는 "뜯어낼 수 있는(rippable)" 하네스를 구축하라.

### 2. 정적 하네스 설계
하네스는 모델 발전에 따라 진화해야 한다. 새로운 추론 능력은 기존 최적화 미들웨어를 역효과로 만들 수 있다. 주요 모델 릴리즈에 맞춰 하네스 컴포넌트를 업데이트하라.

### 3. 문서화 소홀
가장 큰 영향을 주는 하네스 개선이 종종 문서 강화다. 모호한 AGENTS.md는 모호한 에이전트 출력을 낳는다. 에이전트의 그라운드 트루스가 되는 정밀하고 기계 판독 가능한 문서에 투자하라.

### 4. 피드백 메커니즘 부재
피드백 없는 하네스는 가이드가 아니라 감옥이 된다. 에이전트는 다음을 통해 성공/실패를 알아야 한다:
- 태스크 완료 전 자가검증
- 에이전트 워크플로 내 테스트 실행
- 태스크 카테고리별 성공률 메트릭

### 5. 인간 전용 지식 저장
사람들의 머릿속이나 접근 불가능한 문서에 있는 아키텍처 결정은 에이전트에게 갭을 만든다. 에이전트에게 필요한 모든 것은 리포지토리 안에 있어야 한다.

## 하네스 엔지니어링 vs. 관련 분야

| 분야 | 범위 | 주요 초점 |
|------|------|----------|
| **프롬프트 엔지니어링** | 단일 상호작용 | 효과적인 프롬프트 작성 |
| **컨텍스트 엔지니어링** | 모델 컨텍스트 윈도우 | 모델에 제공되는 정보 |
| **하네스 엔지니어링** | 전체 에이전트 시스템 | 환경, 제약, 피드백, 생명주기 |
| **에이전트 엔지니어링** | 에이전트 아키텍처 | 내부 에이전트 설계 및 라우팅 |
| **플랫폼 엔지니어링** | 인프라 | 배포, 스케일링, 운영 |

하네스 엔지니어링은 컨텍스트 엔지니어링을 포함하고 프롬프트 엔지니어링에서 일부를 가져오지만, 더 높은 시스템 레벨에서 작동한다.

## 엔지니어 역할의 변화

### 필수 스킬로서의 하네스 엔지니어링

| 전통적 역할 | 신흥 역할 |
|------------|----------|
| 코드 작성 | AI가 코드를 쓰는 환경 설계 |
| 코드 디버그 | 에이전트 행동 패턴 디버그 |
| 코드 리뷰 | 출력물 + 하네스 효과성 평가 |
| 테스트 작성 | 에이전트를 위한 테스트 전략 설계 |
| 문서 유지 | 기계 판독 가능한 인프라로서의 문서 구축 |

### 하네스 엔지니어 핵심 역량

1. **시스템 사고**: 제약, 피드백 루프, 문서 간의 상호작용 이해
2. **아키텍처 설계**: 강제 가능하고 생산적인 경계 수립
3. **명세 작성**: 자율 실행이 가능할 만큼 정밀하게 의도 표현
4. **옵저버빌리티**: 에이전트 행동 패턴을 드러내는 모니터링 구축
5. **반복 속도**: 하네스 구성을 빠르게 테스트하고 개선

---

## Ashby의 법칙 (필수 다양성의 법칙)

> 조절자(regulator)는 자신이 지배하는 시스템만큼의 복잡성을 가져야 한다. 즉, 모델링할 수 있는 것만 규제할 수 있다.

**적용**: LLM 기반 에이전트는 거의 무엇이든 생성할 수 있지만, 정의된 토폴로지(서비스 템플릿, 아키텍처 패턴)에 커밋함으로써 가능성 공간을 좁혀 포괄적인 하네스 구축이 가능해진다. 이는 **다양성 축소 전략**으로 거버넌스 가능성을 높인다.

## 핵심 요약

1. **하네스 엔지니어링**은 제약, 피드백 루프, 문서, 생명주기 관리를 통해 AI 에이전트의 신뢰할 수 있는 동작을 가능케 하는 새로운 규율이다.

2. **기반 모델은 상품**이다. 경쟁 우위는 하네스 설계에서 나온다 — LangChain은 하네스 최적화만으로 Top 5 벤치마크 성능을 달성했다.

3. **OpenAI는 프로덕션 규모의 실현 가능성**을 증명했다: 수동 코드 없이 100만 줄 이상.

4. **세 가지 기반 축**: 컨텍스트 엔지니어링, 아키텍처 제약, 엔트로피 관리.

5. **단순한 구현**(잘 설계된 AGENTS.md, 프리커밋 훅)이 복잡한 미들웨어보다 더 큰 영향을 줄 때가 많다.

6. **엔지니어의 역할**은 코드 작성에서 AI가 자율적으로 코드를 생성하는 환경을 설계하는 것으로 진화하고 있다.

7. **적응 가능한 하네스를 구축**하라 — 모델 발전에 따라 망가지는 과도하게 엔지니어링된 시스템은 장기적 가치를 떨어뜨린다.
